I/O Planning and Placement

Introduction to IO Planning

The IO placement engine consists of set of commands by which user can optimize IO position. It is a row based placement methodology. The first step in IO planning and placement is to create IO rows. Once the IO rows are created, then place PADs inside the IO rows. After PADs are placed inside the IO rows, you can insert filler cells in the gaps between the pads and the rows. Thereafter, you can place corner cells on the 4 corners of the design.

The following commands have been provided for IO placement engine in the Virtuoso platform:

  • Row Creation
    Creates IO rows of PAD type siteDefs available in technology.
  • PAD Placement
    Places celltype of PAD instances in IO rows.
  • Filler Cells
    Inserts filler cells once Pads instances are placed to maintain signal continuity in the IO row.
  • Corner Cells
    Inserts corner cells not in the row but in between the rows to maintain signal continuity between IO ring.

Database Preparation

In database preparation, IO Placer requires information about the technology file, Site pattern, and cell type for pads, fillers, and corners.

In technology file, ensure that at least one SiteDefs exists and is of PAD type.

Site pattern - for each instance, that is, pad, corner, filler, you can define a site pattern. A site pattern is a set of sitedefs. The site definition is read from the technology file. For more information on how to set site definition, refer to the Virtuoso Technology Data ASCII Files Reference user guide.

Row Creation

In Floorplanner, the first step that IO planning and placement has is to create IO rows. The rows determine the legal placement area for pads, where to place the PADs and the type of PADs that are to be placed. You can manually create rows and use IO Placer with manually created rows. However, the manually created rows need to have the right settings. There is no restriction on where should the rows be created. Also, it is not mandatory to create rows on the periphery.

To create rows, you can click the Floorplan – I/O Placer - I/O Row Create command. The Create IO Row form is displayed.

(ICADVM20.1 Only) In Layout EXL, choose Plan – I/O Planning – I/O Row Create.
The site definition of the type ‘PAD’ will be read from the technology file and displayed in the Site Pattern fields. If there is no site definition of type PAD available, the row creation UI is not displayed.

In the Create IO Row form, the Rows Per Side section has two options, One Row and Two Rows. One Row is selected by default and the options for Row 1 are displayed. A maximum of two rows can be created on each side (i.e. TOP, LEFT, BOTTOM, and RIGHT) of the top level boundary. If you want to create two rows on each side, select Two Rows. This will insert the UI for the second row, as displayed in the following figure

To create a row, you need to specify the row site patterns and their orientation for each side from the Site Pattern and Site Orient combo fields, respectively and then click OK. This will create the row at the appropriate position. If the row already exists at the position where new row is to be created, the existing pad rows will be replaced by the new row. If you run this command again and some rows already exists, they will be deleted automatically.

The site pattern is extracted from the technology file. The site orientation is the relative orientation of the pad inside the row and not the final pad orientation.

PAD Placement

PAD placement is an essential part of the IO placement methodology. It is the next step after row creation in IO Placer. IO rows should be present before running PAD placement. The PADs are placed on the periphery of the design in the IO rows.

To place PADS on IO rows, click the Floorplan – I/O Placer – I/O PAD Placer command. The IO PAD Placer form is displayed.

(ICADVM20.1 Only) In Layout EXL, choose Plan – I/O Planning – I/O Pad Placer.

The IO PAD Placement form has two sections — Style and Spacing. When the IO Placer form is invoked, you can select the placement style and put the placement constraints if required.

Style

There are two types of placement styles that are supported, Connectivity Based and Non-Connectivity Based. The Style section allows you to select either ‘Connectivity Based’ or ‘Non Connectivity Based’ style of placement. By default, Connectivity Based style is selected.

  • Connectivity Based
    You can use this option to place the PADs in the rows minimizing the net length between the PADs and the placed blocks in the design.
    The assumption is that in Connectivity mode, the core instances re placed inside the prBoundary.
  • Non Connectivity Based
    This option enables you to distribute the PADs around the periphery of the design boundary without considering the connectivity between PADs and blocks placed in the design.

Spacing

Spacing constraints defines the inter spacing between the PADs in a given row. This is a post processing step for the PAD placement engine. To define the inter spacing between PADs on per row basis, you can select any one of the following four spacing constraints

  • None
    This option is selected by default. This means that all the placement constraints are OFF by default and pads will be placed with spacing value 0.
  • Even
    You can select this option to distribute PADs evenly (with same spacing between PADs) in the designated row.
  • Abut
    This option guides the placement engine to abut the PADs in the designated row. The selection of abutment will depend upon the row side.
    • If the row is horizontal (TOP or BOTTOM), then the choices are ‘LEFT’, ‘RIGHT’, and ‘CENTER’.
    • If the row is vertical (LEFT or RIGHT), then the choices are ‘TOP’, ‘BOTTOM’, and ‘CENTER’.

    For example you may specify area as Left, Right, and Center for the top and bottom rows and Top, Bottom, and Center for the left and right rows.

  • Alignment Constraint
    You need to specify the side (TOP, BOTTOM, LEFT, RIGHT) for pad instances.
    IO Placer preserves the order in which the pad instances are placed in the design. In some cases, you may need to set the PreserveOrder parameter of the alignment constraint to TRUE to get the required results.
  • Orientation Constraint
    You need to specify the orientation for pad instances.
  • Spacing constraints
    The spacing constraint is defined in the IO Pad Placement form (Even, Abut, and Custom).
  • Placement Blockage
    I/O Pad Placer respects placement blockages and pre-placed instances and does not allow you to place an instance in the design. This means while placing a PAD, if an instance is found on a location where the PAD is to be placed, the placement of the PAD is accordingly adjusted leaving the instance as is.

Filler Cells

The Filler cells insertion is needed to maintain signal continuity in the IO ring for power and ground distribution and for ElectroStatics Discharge (ESD) protection reasons.

Filler cells insertion is done after PAD placement is done in the IO rows of the design. The filler cells are inserted in the empty space between PADs in an IO row. In order to fill the gap, the filler instances with the exact height matching the row site height are chosen. This is done to minimize the number of filler cells that are inserted by the insert filler engine. The filler cells that are to be inserted will be used from the Filler Cells table in the Insert Filler Cells form.

To insert filler cells, you first need to define the design area. It could either be an entire design or a selected region of a design. After defining the design area, select the filler cells from the libraries that are to be used.

To open the Insert Filler Cells form click the Floorplan – I/O Placer – Insert Filler command. The Insert Filler Cells form is displayed.

(ICADVM20.1 Only) In Layout EXL, all Floorplan commands are available under the Plan menu.

The Insert Filler Cells form has three sections.

  • Region Selection Mode
  • Select Filler Cells
  • Filler Instance Attribute

Region Selection Mode

In this section, you can select either of the two modes, i.e, work on entire design or work on a selected region.

  • Entire Design
    Enables you to select the entire design for inserting filler cells. This means that filler cells will be inserted in all the IO rows of the design.
  • Selected Region
    Enables you to select a region for inserting filler cells. This means that only few IO rows need to be used for filler insertions.
    • Draw
      Clicking Draw will enable you to select the area of the design in the canvas, on which operation needs to be performed. This will automatically populate the Coordinates field accordingly.
    • Coordinates
      Defines the coordinates of the selected region. You can manually edit the value in the coordinates.
    • Zoom to
      Clicking the Zoom to button will zoom in the selected region.

Select Filler Cells

The Select Filler Cells section facilitates you in selection of few cells from entire gamut of libraries/cells/views. The few selected cells will be used by the engine as filler cells.

To select a lib:cell:view for the filler cell, do one of the following:

  • Type the library, cell, and view names in corresponding fields.
  • Click Browse to display the Library Browser window. Select the lib:cell:view name from the window. The value of the selected lib:cell:view name will be displayed in the Library Name, Cell Name, and View Name fields, respectively. You can keep the Library Manager window open and click Add Filler as soon as a new cell is selected.
    • Add Filler
      To add the selected lib:cell:view name as a filler, click Add Filler. The selected lib:cell:view name will be displayed in the filler table.
      Invalid lib:cell:view values are not added to the filler table. An appropriate warning message is displayed in the CIW.
    • Remove Filler
      To delete the selected lib:cell:view name as a filler, click Remove Filler. It will just remove the selected rows from the table.

Filler Instance Attribute

In this section, you can define the instance name prefix and the placement status of the filler cell.

  • Instance Name Prefix
    Contains the instance name prefix, which will be pre-pended to the inserted physical only filler instances. This will help you recognize the filler inserted in a specific row or location for deleting them or doing some edits.
    The physicalOnly attribute is automatically added to the new cells that are inserted.
  • Mark Fillers as Fixed
    Select this field to specify the placement status of inserted cells as Fixed. By default, the check box is not selected. When the check box is deselected, the placement status of the inserted cells is set to Placed.

After clicking OK or Apply in the Insert Filler Cells form, the filler will be inserted according to their site pattern (to choose the correct row to place them) and then filler cells will be placed from the biggest to the smallest in order to minimize the number of inserted filler cells.

Filler cells with height less than row site height will not be considered. This is because if we have filler cells lesser than the row site there will be discontinuity in the rails.

Corner Cells

Corner cells insertion is needed to maintain signal continuity in the IO ring between the IO row power and ground distribution. The four corner cells are required to be placed on the four corners of the design.

Corner cells insertion and deletion is done after PAD and filler cell placement is done in the IO rows of the design.

The high level algorithm for corner cells is as follows:

  • The list of the selected instances are filled in the corner UI.
  • Engine takes the inputs instance name and its orientation for each corner from the UI.
  • The corner row is created of appropriate site(s) in the design before placing the corner cells.
  • Corner cells will be placed in the appropriate row as per the orientation specified.

Before inserting corner cells, ensure the following:

  • The library has at least four instances with their cellType set to corner. If not, a suitable message is displayed in the CIW.
  • The row in which the corner cells must be placed has the same sitePattern as the corner cells. Corner cells can be placed only in rows with matching sitePattern.
  • The technology file contains the sitePattern under the siteDefs section. If not, a suitable message is displayed in the CIW.

If all the above conditions are met, a corner row of the given sitePattern is created, and cells with cellType corner are inserted in these rows.

The procedure to insert corner cells is as follows

  • Selecting corner instances
    Ensure that the corner instances are already present in the design. The corner instances can be selected in the following ways:
    • Navigator
    • Search Engine
    • Canvas Selection

Once the corner instances are selected in the design, invoke the Corner Pad Placer command by clicking Floorplan – I/O Placer - Corner Pad Placer. The Corner Pad Placement form is displayed. The form is by default populated with the selected instances. However, if the corner instances are not selected, the form will not be displayed and a corresponding message is displayed in the status bar.

  • Placing the corner - In the Corner Pad Placement form, you can specify another instance for each corner from the Instance Name list. You can also set the orientation for each corner from the Orientation drop-down list box. Once the instance is selected and the orientation is set, click the OK button. A row will be created automatically. Simultaneously, a corner cell is placed in that row.
