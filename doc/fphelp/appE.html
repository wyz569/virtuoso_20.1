Pin Constraints File

A pin constraints file is a simple SKILL file with constraints entered in a SKILL list. The constraints are entered in a tabular format so that they are readable and editable.

The SKILL list in the pin constraints file includes:

  • Pin Name
  • Edge Constraint
  • Pitch Constraint
  • Status Constraint

The pin constraints file also includes:

  • An API to load the constraints in the Constraint Manager.
  • A Sanity Checker that parses the pin constraints file and reports errors, if any.

Optimized Creation of Pin Constraints

The creation of pin constraints has been optimized such that only one alignment constraint is created for a given side and pitch value. Further, for each design, a single fixed and locked constraint is created.

Pin Name

Pin name identifies the pins for which constraints need to be applied. Valid formats for specifying pin names are listed below:

Format    

Description    

Examples

Regular expressions

Use regular expressions to specify pin names. Wildcard characters and hierarchy delimiter “/” are also recognized.

Format:

'[ Instance_Name / ] Terminal_Name  [ : Pin_Name ] [ : PinFig_Name ]" | "[ Instance_Name /  ] " .* "

Regular expressions with the “<“ and “>” characters are not recognized. Only use the “:” character to specify a range.

'("A:A" "left"   0.01 "locked")

'("A/.*" "left"   0.01 "locked")

The following expression will not be recognized:

'("X<.*"   "right" 0.1 "floating" )

'("X<[0-1]>"   "right" 0.1 "floating" )

'("X<[0-1]>:X<[0-2]"   "right" 0.1 "floating" )

Numeric Pins

This method is used for pins with numeric names.

'("3.1:4.1" "right"   0.1 "floating" )

Terminal names and pin names

Terminal and pin names are named OA objects. Therefore, you can assign constraints to:

  • Terminal names
Constraints assigned to terminal names are applicable to all the pins and pinFig objects corresponding to the terminal.
  • Terminal names and pin names
Constraints are applicable only to the specified pin in the specified terminal.
Format:
'("Terminal_Name:Pin_Name" "Pin_Edge" Pin_Pitch "Pin_Status")

'("A" "Top"   0.01 "locked")

'("A:A1" "Top"   0.01 "locked")

'("A:A2" "Top"   0.01 "locked")

Individual PinFigs

This method is used for pin figures that are placed together to implement a feed through connection. You can enter constraints corresponding to each pin shape.

Format:

'("Terminal_Name:Pin_Name:PinFig_Name" "Pin_Edge"   Pin_Pitch "Pin_Status")

'("A:A:A1" "left"   0.01 "locked")

'("A:A:A1.1" "right"   0.01 "locked")

Pin names for level-1 pins

When you specify instance pins, they correspond to the instance master’s Terminal_Name:Pin_Name:PinFig_Name. Therefore, ensure that you specify the instance name while defining constraints for level-1 or soft block instance pins.

Format:

'("Instance_Name/Terminal_Name:Pin_Name:PinFig_Name" "Pin_Edge"   Pin_Pitch "Pin_Status")

'("I1/A:A:A1" "left"   0.01 "locked")

Wildcard characters and hierarchy delimiters

To specify the constraints on all terminals and pins for a given instance, enter only the instance name and a ".*" after the hierarchy delimiter "/".

To specify the constraints on the top-level pins, enter ".*".

You can use ".*" as a wildcard in regular expressions for names.

You can also specify regular expressions in pin or pinFig names.

'("I1/.*" "left"   0.01 "locked")

'(".*" "left"   0.01 "locked")

'("ADDRESS_.*" "left"   0.01 "locked")

'(".*:PIN246.*" "left"   0.01 "locked")

'(".*:PIN246.*:PinFIG1.*" "left"   0.01 "locked")

'("D[1-3]:P.*"   "right" 0.1 "floating" )

Vector single and vector bit notations

You can include vector single and vector bit notations A<0:4>:A<0:4> in pin names. Such vector notations are automatically expanded to their corresponding vector bits.

'("A<0:4>:A<0:4>" "Top"   0.01 "locked")

If there are duplicate entries for a pin, only the last entry is honored. All previous entries are ignored.

Edge Constraint

The edge constraint specifies the side on which the pin should be placed. Valid values for this constraint are:

“left” | “right” | “top” | “bottom” | “edge_number” | “any”

For shapes with rectangular PRBoundaries, the "left", "right", "top", and "bottom" strings are supported. For shapes with rectilinear PRBoundaries boundaries, specify either the edge side or the "edge_number". Edge numbers are assigned in the increasing order starting with zero, which is assigned to the edge at the right-top vertex, followed by the adjoining edge in the anti-clockwise direction.

An edge side for rectilinear boundaries may correspond to one or more edges. In the following example, the "Top" edge side can correspond to edge numbers 0 or 2. Therefore, the pin planner has a choice of edges.

The "center" and "origin" side parameter values of alignment constraints are not supported because they are not respected by the Pin Optimizer.

Use the "any" side specification for a pin that can be placed on any available edge. When the "any" side is specified, the “orderOnly” option is automatically turned on. Therefore, pin order is respected while placing the pins.

In case of pins with a single pinFig, the alignment constraint is created on the pin, and not on the pinFig—even if the constraint is specified on the pinFig.

Pitch Constraint

The pitch constraint defines the minimum edge-to-edge spacing between pins and other components. The format for specifying the pitch is as follows:

Unsigned Number > mfgGridResolution  | -1

Pitch values greater than the manufacturing grid resolution are valid. If you do not want to specify any pitch value, enter -1 in the table. This will be treated as the pitch value not being set.

During pin optimization, "floating" and “placed” pins are positioned such that the pitch of fixed/locked pins is honored. For more information about these pin statuses, see Status Constraint.
If different pitch values are defined for a pin and a corresponding pinfig, then the pinFig pitch value is respected.

The following examples illustrate how the pitch value is calculated in different situations.

Example 1:

This is a simple example, where the pitch for pin A is 0.3. A pitch can have value zero. Assumption is that the pitch for the neighboring pins or objects is zero. Pins are placed such that adjacent pins or objects are separated by a minimum distance of 0.3, which is the pin pitch value.

Pins A and B are placed at a distance of 0.5, which is the higher pin pitch value.

If bus pins are assigned different pin pitch values, then the values are honored without breaking the bus order or splitting the group.

Status Constraint

The status constraint defines the placement status of the pin. Valid values are "fixed"/"firm", "locked", and "floating".

Pins with the "fixed" or "firm" status can be moved by interactive commands. The "locked" status indicates that the corresponding pin cannot be moved by any commands. The placement of such pins should not be changed. Specify "floating" for pins with no status.

Sanity Checker

The pin constraints file integrates a simple sanity checker to perform a basic sanity check of all the pin names and constraints that are included in the pin constraints file. If there are erroneous entries, then the Pin Optimizer does not respect the constraint. The sanity checker helps ensure that there are no erroneous entries in the pin constraints file.

The sanity checker performs the following checks for pin names:

  • Validity of the naming format
  • Existence of the specified instance, terminal, pin and pinFig
  • Non-existence of duplicate entries for a pin name
    The sanity checker only reports the existence of duplicate entries. It does not perform additional checking to check the constraints.
  • Existence of multiple instances of the same master
    The sanity checker will only report such instances.

In addition, the sanity checker performs the following checks to ensure correctness of the constraints:

  • Validity of the boundary edge in case of a rectilinear prBoundary
  • Validity of the boundary side
  • Checking whether the pin pitch has a higher value than the manufacturing grid or has the value of -1
