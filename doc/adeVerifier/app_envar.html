Verifier Environment Variables

This appendix describes the environment variables of Virtuoso ADE Verifier (Verifier) that you can use to customize the default settings. You can set the values of these environment variables in the .cdsenv or .cdsinit file. You can also set the environment variables from Virtuoso CIW for a session.

The following example illustrates how an environment variable is set in the .cdsenv file. For more details, see Specifying Environment Settings in Virtuoso Design Environment User Guide. 

verifier.preferences    createbatchlogfile string "YES"

The following example illustrates how an environment variable is set in the .cdsinit file or Virtuoso CIW.

envSetVal("verifier.preferences" "createbatchlogfile" 'string "YES")
Only the environment variables documented in this section are supported for public use. Any other environment variables and undocumented aspects of the environment variables described below are private and subject to change or removal at any time.

This appendix describes the following Verifier environment variables:

General Environment Variables

reqspecs

Sets the search order to identify the specifications that Verifier must use.

Variable Group

verifier.preferences

Variable Type

String

Default Value

"Use Requirement Spec and Check" "Use Requirement Spec" "Use Implementation Spec"

Acceptable Values

String

The string value can be:

  • "Use Requirement Spec and Check": Check if the specification values set in the requirements match the specification values set in the corresponding implementations. If the values match, Verifier uses those values to determine the verification status of the requirement. Otherwise, the specification check for that requirement fails.
  • "Use Requirement Spec": Use the requirement specification. If it is missing, use the implementation specification.
  • "Use Implementation Spec": Use the specification values set in the implementations to determine the verification status of the mapped requirements. If an implementation does not have any specification value, use the values set in the requirement mapped to that implementation.

Related Field

Specifications Search Order

tagsToOverrideWithVerificationSpace

When you select the “Local Run" or "Batch with Space” run modes and run simulations, Verifier merges the corners and sweeps in the active setup with corners and sweeps defined in the Verification Space. To ensure that only verification space is considered during simulation (overriding already defined corner and sweeps in active setup), you can set the environment variables as follows: 

envSetVal("verifier.preferences" "tagsToOverrideWithVerificationSpace" 'string "corners sweeps")

To reset the variable to the default behavior:

envSetVal("verifier.preferences" "tagsToOverrideWithVerificationSpace" 'string " ")

Example:

Suppose you have an active setup with corners C0,C1,C2 and a verification space with corners C0,C1,C3.

Here, tagsToOverrideWithVerificationSpace set to "corners" overrides the corners from the active setup. It runs C0, C1, C3 from the verification space, and does not run corner C2.

The default behavior of tagsToOverrideWithVerificationSpace is merging, and not overriding, the corners from the active setup and the verification space. It will merge the corners C0, C1, C2, and C3.

In .cdsenv:

 

verifier.preferences tagsToOverrideWithVerificationSpace string ""

In .cdsinit or the CIW:

 

envSetVal("verifier.preferences" "tagsToOverrideWithVerificationSpace" 'string "corners sweeps")

Variable Group

verifier.preferences

Variable Type

String

Default Value

""

Acceptable Values

"corners" "sweeps""corners sweeps"

Specify the tags to override as a String value.

disableReadOnly

Specifies if Verifier should run the implementations in read-only or write mode. By default, Verifier runs the implementations in read-only mode to ensure that the active setup is not changed by Verifier. If this environment variable is set to t, for GUI mode, you are prompted to check out the implementation cellview before running simulation. For batch mode, Verifier checks out the implementation cellviews directly before running the simulation.

Cadence recommends that you run the implementations in read-only mode. Running implementations in write mode can cause issues like the deletion of history items.

In .cdsenv:

 

verifier.run disableReadOnly boolean nil

In .cdsinit or the CIW:

 

envSetVal("verifier.run" "disableReadOnly" 'boolean t)

Variable Group

verifier.run

Variable Type

Boolean

Default Value

nil

Acceptable Values

nil / t

Related Topic

“Simulating Implementations”
You can view the results from the read-only history. For this, right-click the implementation and choose View Implementation Results.
When this variable is set as below:
envSetVal("verifier.run" "disableReadOnly" 'boolean t)

The implementation run fails in the following conditions:

  • When you run implementation in editing mode from Verifier but the maestro view is already opened in Assembler;
  • When you run two implementations parallelly in editing mode from Verifier and they use the same maestro view with different histories. You can start and run one of these implementations successfully, but the second implementation fails because the same maestro view is being run in two different sessions.

additionalProjectDirectory

Creates the additional project directories.

You can specify multiple paths as a comma-separated list of values. If a path contains special characters, such as white spaces, you do not need to escape the special characters by using the '\' character. For example, for specifying /path/with/white\ space/, you can specify /path/with/white space/.

To add a project directory, you must specify a non-empty project directory while ensuring that the directory path is readable and valid. For this, you can also use the UNIX environment variables ($) and home directory (~).

For path including white spaces (such as "ab c"), you do not need to escape the character ' '.
