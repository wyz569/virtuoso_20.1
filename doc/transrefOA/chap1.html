Overview of the Translators

This chapter contains the following sections:

  • Overview
  • Understanding the Translators
  • Running the Translators

Overview

The Cadence® Virtuoso® Design Environment is the foundation on which most Cadence tools are built. The Virtuoso Design Environment on OpenAccess is an open system. You can enter your own design data with industry-standard data formats and use the Virtuoso Design Environment tools to complete your job.

The design data translators let you translate a design in one format to a Cadence database or to take a Cadence database and translate it into another format. Cadence offers a variety of translators to use in integrating your work with the OpenAccess database.

Understanding the Translators

The design data translators fall into three categories: general, netlist, and physical translators.

The general translators translate EDIF and Verilog files to and from OpenAccess database. The netlist translators translate netlist designs to and from OpenAccess database. The physical translators translate physical (layout) designs to and from OpenAccess database.

General Translators

The general translators are:

  • EDIF 200 Writer, which translates designs from OpenAccess database to EDIF 200 format. For information about EDIF 200 Writer, see the Virtuoso EDIF 200 Reader and Writer User Guide.
  • Verilog In, which imports a Verilog HDL description to a OpenAccess database. For information about Verilog In, see the Verilog In for Design Framework II User Guide and Reference.

Netlist Translators

The netlist translators are:

  • Spice In, which translates SPICE and Circuit Definition Language (CDL) files to OpenAccess database. For information about Spice In, see Chapter 4, “Netlist Import Using Spice In”.
  • CDL Out, which translates a OpenAccess schematic design to a hierarchical netlist in Circuit Definition Language (CDL) format. For information about CDL Out, see Chapter 5, “Design Translation Using CDL Out”.

Physical Translators

The physical translators are:

  • Stream In, which translates Graphics Design Station (GDSII) Stream mask data from Stream format to the OpenAccess database. For information about Stream In, see Chapter 2, “Design Translation Using XStream Translator”.
  • Stream Out, which translates files from OpenAccess database to Graphics Design Station II (GDSII) Stream mask data format. For information about Stream Out, see Chapter 2, “Design Translation Using XStream Translator”.
  • LEF In, which translates a LEF file to an OpenAccess technology library and macro information in the OpenAccess database. For more information on LEF In, see Chapter 6, “Design Translation Using LEF/DEF Translator,”
  • LEF Out, which translates an OpenAccess technology library and macro information to a LEF file. For more information on LEF Out, see Chapter 6, “Design Translation Using LEF/DEF Translator,”
  • DEF In, which reads a DEF file and creates an OpenAccess cellview. For more information on DEF In, see Chapter 6, “Design Translation Using LEF/DEF Translator,”
  • DEF Out, which writes a DEF file from an OpenAccess cellview. For more information on DEF Out, see Chapter 6, “Design Translation Using LEF/DEF Translator,”

Running the Translators

You can run all the design data translators in interactive mode by using the user interface forms present in the translators. You can access these forms by invoking the appropriate menu options available in the CIW. You can also run most of the translators in batch mode by entering a command on the command line.
